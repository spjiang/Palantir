# 应急安全AI操作系统（城市暴雨内涝指挥）PPT（基于 V7）

目标：把“暴雨内涝指挥”从一次性项目，做成可复制、可装配、可治理的行业操作系统能力。

---

## 1. 对标 Palantir：三条业务线能力在应急行业的落点

- **Foundry（数据语义与装配）**：把多源异构数据变成可复用的“数据产品 + 语义对象”（统一口径、可治理、可追溯）
- **Gotham（任务指挥与行动）**：把态势与行动闭环（任务、审批、协同、证据链、时间线）
- **Apollo（持续交付与治理）**：把规则/策略/模型/知识的变更变成可控发布（版本化、回测、回滚、审计）

> 本 PPT 以 V7 的 5 步闭环“业务流程节点”为主线，逐节点讲清楚：每一步**做什么、为什么复杂、怎么保证专业与可控**。

---

## 2. 业务闭环总览（V7 1.1）

- 1. **风险热力图滚动更新（未来 1–3 小时）**
- 1. **暴雨参谋长智能体提示 TopN 高风险点位（原因 + 置信度）**
- 1. **一键生成“布防/封控/联动”任务包并下发（高风险动作必须人审）**
- 1. **移动端回执与证据回传，指挥端实时显示**
- 1. **一键生成战报（时间线 + 指标 + 依据）**

> 每个节点都要求：可解释（证据/置信度/版本）+ 可闭环（任务/证据/回执）+ 可追溯（血缘/审计）。

---

## 3. 节点 1：风险热力图滚动更新（未来 1–3 小时）

### 3.1 这个节点“做什么”

- **把降雨/水位/泵站/路况/易涝点等多源数据**，转成统一可视的“网格/路段风险态势”（热力图 + TopN）
- 输出要能直接支撑：**预警分级、指挥调度、资源前置布防**

### 3.2 输入 / 处理 / 输出（业务口径）

- **输入（态势要素）**：
  - 短临预报/雷达回波（0–2h）
  - 雨量站（分钟级累计雨量/雨强）
  - 水位站/闸口水位（趋势、汛限/警戒水位）
  - 泵站工况（启停、流量、故障码、维护窗口）
  - 路况（拥堵/封闭）
  - 台账：易涝点、下凹立交、排水能力、责任片区、重点目标
- **处理（关键动作）**：
  - 事件时间对齐（避免乱序/迟到）
  - 空间映射（网格→汇水区/道路；雨量站校准/修正）
  - 窗口聚合（5/15/60 分钟）
  - 风险推理（网格/路段 risk_score/risk_level）+ 置信度/解释因子
- **输出（指挥可消费）**：
  - 风险热力图（网格级）
  - TopN 风险路段/点位（含原因、置信度、版本引用）

### 3.3 为什么复杂（行业专业点）

- **短临误差与不确定性**：预报随时间拉长误差增大，必须把不确定性体现在置信度
- **汇水区/产汇流机理**：雨不等于涝，地形低洼、不透水率、管网能力决定积水形成速度
- **顶托/倒灌**：外河水位高导致外排受限，内涝会快速恶化
- **设备维护窗口**：泵站维护/切换工况会造成“假异常”，需要数据质量与抑制策略

### 3.4 对标能力映射

- **Foundry**：统一事件封装 + 标准事实层 + 语义对象状态（可装配的态势数据产品）

## 4. 节点 2：暴雨参谋长智能体提示 TopN 高风险点位（原因 + 置信度）

### 4.1 这个节点“做什么”

- 把“风险 TopN”升级为“可决策建议”：
  - **为什么是它（原因）**
  - **有多确定（置信度）**
  - **需要补什么证据（核验清单）**
  - **对应预案条款/处置规程（依据引用）**

### 4.2 输入 / 处理 / 输出

- **输入**：
  - 节点 1 输出（风险/置信度/解释因子）
  - 本体对象状态（路段/泵站/地铁口/重点目标）与关系（locatedIn/near/serves/affects）
  - 行业知识：预案/规程/条款/历史战报（用于证据引用）
  - 数据质量 DQ（新鲜度/有效性/一致性）
- **处理**：
  - 检索增强生成（先过滤再检索）：按区域/对象类型/响应级别/动作类型检索条款
  - 证据门槛：涉及封控/停运等必须同时具备“条款证据 + 数据证据”
  - 结构化输出：建议清单 + 任务草稿 + 证据引用 + 风控约束
- **输出**：
  - TopN 研判摘要（含置信度解释）
  - 处置建议（动作、对象、优先级、前置条件）
  - 任务包草稿（owner_org/SLA/required_evidence/need_approval）

### 4.3 为什么复杂（行业专业点）

- **“预警≠处置”**：同样的风险等级在不同区县/不同道路等级/不同交通时段，处置口径不同
- **证据与责任必须对齐**：建议必须能落到责任片区与部门边界，否则无法执行
- **避免幻觉**：没有证据就必须输出“无法判定 + 核验清单”，不能编造

### 4.4 对标能力映射

- **Foundry**：语义对象与证据检索的装配（数据产品 + 预案条款库）
- **Gotham**：把态势翻译成可行动的“任务草稿”

---

## 5. 节点 3：一键生成“布防/封控/联动”任务包并下发（高风险动作必须人审）

### 5.1 这个节点“做什么”

- 将研判建议变成可执行的**任务包**，并通过审批/派单进入跨部门协同
- 特别强调：**封控/停运/跨部门联动**等高风险动作，必须进入“人审确认”节点

### 5.2 输入 / 处理 / 输出

- **输入**：
  - 节点 2 的任务草稿（tasks[]）
  - 组织与责任体系（区县/街道/排水/交警/地铁等）
  - 风控策略（哪些动作必须审批、哪些动作需要二次确认）
- **处理**：
  - 任务包编排：任务类型标准化（巡查/排涝/封控准备/联动通知…）
  - 派单路由：按区域 + 对象类型 + 责任片区路由到单位/岗位
  - 人审门禁：need_approval=true 的任务必须走审批；审批结论回写时间线
- **输出**：
  - 已下发任务（任务状态机起点）
  - 审批记录（通过/驳回/补充核验）

### 5.3 为什么复杂（行业专业点）

- **跨部门协同成本高**：排水/交警/地铁/城管/街道联动的“边界与口径”必须固化
- **高风险动作治理**：封控对社会影响大，必须“可解释 + 可追责 + 可回滚”
- **SLA 与升级**：暴雨场景时效性强，超时必须催办/转派/升级上报

### 5.4 对标能力映射

- **Gotham**：任务体系（审批、派单、状态机、协同）
- **Apollo**：策略版本化（审批策略、派单路由策略可控变更）

---

## 6. 节点 4：移动端回执与证据回传，指挥端实时显示

### 6.1 这个节点“做什么”

- 把“执行结果”结构化回传：到场、处置动作、结果、证据附件
- 指挥端把任务进度与证据链回灌到态势与时间线，形成真正闭环

### 6.2 输入 / 处理 / 输出

- **输入**：
  - 已下发任务（任务类型、目标对象、required_evidence、SLA）
  - 现场回执（定位、照片/视频、表单字段：积水深度、封控状态、泵站启停等）
- **处理**：
  - 证据校验：未满足 required_evidence 不允许“完成”
  - 状态推进：pending→accepted→in_progress→done/rejected
  - 超时升级：remind/reassign/escalate
  - 证据链绑定：证据与任务、事件、本体对象建立引用关系
- **输出**：
  - 可审计回执（含证据引用）
  - 指挥端实时态势更新（任务进度、封控状态、处置效果）

### 6.3 为什么复杂（行业专业点）

- **证据口径不统一**：不同任务类型回执字段不同，必须标准化表单与必填证据
- **现场数据不确定**：定位漂移、照片延迟、网络不稳定，需要容错与一致性策略
- **责任追溯**：回执不仅是“完成”，还要支撑复盘与绩效（可追责、可验收）

### 6.4 对标能力映射

- **Gotham**：执行与协同闭环（任务 + 证据链）
- **Foundry**：证据成为数据资产（可检索、可复用、可追溯）

---

## 7. 节点 5：一键生成战报（时间线 + 指标 + 依据）

### 7.1 这个节点“做什么”

- 把一次事件沉淀为“可审计的事件档案”：时间线、指标、证据链、依据引用
- 同时为后续策略/模型/预案优化提供结构化输入（闭环学习）

### 7.2 输入 / 处理 / 输出

- **输入**：
  - 预警事件、分析结果、研判输出、人审记录、派单与回执、证据引用
  - 版本信息：规则/策略版本、模型版本、知识版本、注入元信息（如有）
- **处理**：
  - 时间线汇总（who/when/what/why）
  - 指标计算：提前量、预警→派单时延、派单→到场时延、闭环时长、完成率、证据完备率
  - 依据引用：将关键结论绑定到数据证据/条款证据/版本引用
- **输出**：
  - 战报（概览→态势变化→关键决策与依据→执行效果→指标→改进项）
  - 可追溯清单（lineage/version_refs/inject_meta）

### 7.3 为什么复杂（行业专业点）

- **“讲得清楚”不等于“可审计”**：战报必须能追溯到数据与版本，才能经得起复盘与问责
- **指标口径统一**：不同城市、不同部门口径不一致会导致争议，需要统一指标定义与计算口径

### 7.4 对标能力映射

- **Foundry**：事件档案作为可复用数据产品（可查询、可回放、可对比）
- **Apollo**：把复盘结论回写为新版本策略/规则/知识，并可回测/回滚

---

## 8. 全链路“为什么能交付”：三类硬能力（Foundry/Gotham/Apollo 的交集）

- **语义统一**：对象/关系/责任/证据统一到本体（否则只能“看图”，无法“算与派”）
- **版本治理**：规则/策略/模型/知识必须版本化（否则无法复盘、无法验收）
- **闭环执行**：任务状态机 + 人审门禁 + 证据链（否则无法落地、无法追责）

---

## 9. 验收要点（按 V7 的“可重复、可解释、可闭环”）

- **可重复**：同一回放/注入策略可多次复现关键拐点与派单链路（误差可量化）
- **可解释**：TopN 风险与关键建议必须带“解释因子 + 置信度 + 依据引用（条款/数据）”
- **可闭环**：任务必须有回执与证据；战报能自动汇总时间线与指标
- **可追溯**：抽样预警/任务可追溯到 `lineage + version_refs + inject_meta（如有）`
