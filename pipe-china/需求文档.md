# 管网运维业务方案（暴雨应急与日常运维）

## 1. 背景与目标
- 本方案面向“城市管网日常运维 + 暴雨应急处置”的一体化指挥与协同。
- 典型应用区域：北京市海淀区东北旺西路8号院中关村软件园一期尚东数字谷A区及周边道路/园区。
- 业务目标：
  - 提前发现风险（暴雨导致的倒灌、溢流、积水、泵站超负荷或故障）。
  - 快速联动处置（巡检、排涝、封堵、阀门操作、交通协同），并全程留痕。
  - 事后可追溯复盘（事件时间线、关键动作依据、任务执行证据、指标统计）。

## 2. 业务角色与关切
- 调度/指挥：关注风险热点、影响范围、建议动作、派单进度。
- 运维班组：关注收到的任务、SLA、必传证据、审批要求。
- 领导/值班：关注可解释的研判结论、依据、战报。
- 审计/合规：关注操作留痕、版本、证据链。

## 3. 业务对象（实体）
- 管段（PipeSegment）：名称、地理位置、口径、压力、状态、所属责任区域。
- 阀门（Valve）：名称、位置、当前开闭状态、最近操作时间。
- 泵站（PumpStation）：名称、位置、流量、工况/故障码、服务片区。
- 监测点/传感器（Sensor）：类型（压力/流量/液位/雨量）、位置、实时数值、单位、数据质量标签。
- 雨量站（RainGauge）：位置、近 5 分钟/1 小时雨量、设备状态。
- 报警事件（Alert）：事件类型、关联对象、级别、触发原因、时间。
- 任务（Task）：目标对象、任务类型（巡检/封堵/排涝/复位等）、责任单位、SLA、必需证据（定位/照片/表单）、是否需要审批。
- 事件/工单（Incident）：关联报警、任务集合、状态、牵头单位。

## 4. 业务关系（边）
- SERVES：泵站 → 管段（服务/排涝）。
- LOCATED_IN：对象 → 行政/责任区域。
- FEEDS：雨量站/传感器 → 管段或区域（提供观测数据）。
- AFFECTS：故障/报警 → 任务或事件（触发影响）。
- DEPENDS_ON：管段/阀门 → 上游/下游设备或电力。
- NEAR：空间邻近关系，用于影响范围和风险扩散。

## 5. 业务规则示例
- R1：泵站流量持续下降且下游水位上升 → 创建“巡检泵站”任务；SLA 30 分钟；证据需“照片+现场读数”；审批：否。
- R2：管段压力突降且阀门异常打开 → 创建“关闭阀门”任务；需审批：是；证据需“操作记录+定位”。
- R3：区域雨量达阈值且关键管段多次预警 → 生成“排涝预案”任务包（巡检 + 封堵准备）；关键动作需审批。

## 6. 业务处置闭环（方案叙事）
### 6.1 触发与研判
- 暴雨来临时，雨量站与雨量传感器持续上报；若雨量在短时间内快速上升，系统会提示“区域暴雨风险上升”。
- 同时，泵站、管段压力/流量/液位等监测点持续上报；若出现“流量下降 + 水位上升”“压力突降”等组合征兆，系统提示“疑似倒灌/淤堵/设备故障”。
- 调度中心需要在一张图上看到：风险点位、受影响对象（泵站/管段/阀门/周边道路），以及它们之间的上下游或空间邻近关系。

### 6.2 任务编排与审批
- 对低风险动作（如现场巡检、清掏排水口）可直接派单给运维班组。
- 对高风险动作（如阀门操作、封堵/切换、影响面较大的联动措施）必须先走审批，审批通过后再下发执行。
- 每个任务必须明确：责任单位、SLA、必传证据（定位、照片、现场读数、操作记录）。

### 6.3 执行与回执
- 运维人员到场后按任务要求回传：到场时间、处置动作、结果说明、定位与照片/视频，必要时填报现场读数（压力/液位/流量）。
- 若任务超时或证据不全，调度中心可催办、转派或升级处理。

### 6.4 复盘与战报
- 事件结束后生成战报：包含预警与研判过程、关键决策与审批、任务执行与证据、以及关键指标（响应时长、完成率、超时率）。
